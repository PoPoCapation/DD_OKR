# 开发路线：KR/任务/对齐/复盘功能迭代

> 目标：在现有 OKR_REPLACE 项目基础上，分阶段落地关键结果与任务管理、目标对齐、进度记录、复盘、操作审计以及 Case 编排扩展。以下流程按依赖顺序安排，建议逐阶段推进并在每个阶段结束后完成回归验证。

## 0. 约定与准备
- **分层结构**：沿用 trigger → domain → infrastructure → types 四层。
- **命名规范**：表使用 `okr_` / `sys_` 前缀，PO/VO 与 Mapper 保持一致。
- **数据访问**：MyBatis XML 放在 `OKR_REPLACE-app/src/main/resources/mybatis/mapper/`，Domain 经由 `I*Repository` / `I*Service` 暴露。
- **测试策略**：每完成一个 CRUD 模块至少补 1 个 happy-path 集成测试；对进度计算等纯逻辑补单元测试。

## 1. KR CRUD + 进度汇总
1. **DAO / Mapper**：
   - 扩展 `okr_key_result_mapper.xml`，支持按 objectiveId 查询、批量更新状态。
2. **Repository / Service**：
   - 新增 `IOkrKeyResultRepository`、`OkrKeyResultService`，封装 CRUD。
   - 在 `OkrObjectiveService` 增加 `recalculateObjectiveProgress(objectiveId)`，规则：`objective.progress = Σ(KR.rate * KR.weight) / Σ(weight)`。
3. **Trigger 层**：
   - 新建 `OkrKeyResultController` (`/api/okr/kr`)：`create/update/delete/listByObjective`。
   - 权限校验：当前用户必须在可编辑范围内（owner 或下级）。
4. **测试**：
   - 单测 KR 汇总逻辑。
   - 集成：创建 3 个 KR → progress 汇总 → 更新 / 删除 KR → progress 重算。

## 2. 任务 CRUD + 用户关联
1. **数据表**：`okr_task`、`okr_task_user`。
2. **Repository / Service**：
   - `IOkrTaskRepository`、`OkrTaskService`：CRUD + 按 KR 查询。
   - `IOkrTaskUserRepository`：维护 `assignUsers(taskId, userIds)`。
   - 业务要求：任务必须挂载在当前用户可编辑的 KR 下；状态变更写入进度记录。
3. **Trigger**：
   - `OkrTaskController`：`create/update/delete/listByKr`、`assignUsers`、`myTasks`。
4. **测试**：任务创建 + 用户指派 + “我的任务”查询。

## 3. 目标对齐 Alignment
1. **数据模型**：`okr_objective_alignment`。
2. **服务设计**：
   - `IOkrObjectiveAlignmentRepository`，封装 `create/update/delete/findInbound/findOutbound`。
   - `OkrObjectiveService` 中新增接口：`linkObjectiveToParent`、`unlinkAlignment`。
   - 防环校验：从子向上 DFS，禁止回到自身。
3. **Trigger**：`OkrAlignmentController` 提供创建、解除、查询对齐树接口，返回树形 DTO 给前端。

## 4. 进度记录 Progress Record
1. **数据表**：`okr_progress_record`（target_type/target_id + old/new + operator）。
2. **服务**：`OkrProgressRecordService.recordChange(...)`，在以下场景调用：
   - KR completion rate 更新。
   - Objective progress 重算。
   - 任务状态变化、完成度调整。
3. **查询接口**：按 target / 操作者 / 时间范围分页。
4. **测试**：模拟 KR 更新后是否生成记录；分页是否正确。

## 5. 复盘 Check-in
1. **数据表**：`okr_check_in`、`okr_check_in_item`。
2. **流程**：
   - `OkrCheckInService.createCheckIn(objectiveId, cycleId, userId, summary, risk, blocker, nextPlan, confidence)`。
   - 同时写入 `OkrCheckInItem`，记录每个 KR 变动 (`old_completion_rate` / `new_completion_rate`)。
   - 创建成功后调用 `ProgressRecordService` 记录事件类型 `CHECK_IN`。
3. **权限**：目标 owner 或其上级可复盘。
4. **Trigger**：`OkrCheckInController` 暴露创建 / 查询 / 列表接口。

## 6. 操作日志 Operation Log
1. **数据表**：`okr_operation_log`。
2. **服务**：`OperationLogService.log(module, resourceType, resourceId, action, before, after, operatorId, remark)`。
3. **埋点范围**：
   - KR、任务、对齐、复盘等关键操作。
   - 在 domain 层调用，保证与业务事务一致。
4. **查询接口**：按模块 / 操作人 / 时间过滤，供审计使用。

## 7. Case 编排扩展
1. **现有结构**：`OKR_REPLACE-case` 中已有 OKR 创建/查询流程。
2. **扩展思路**：
   - 新增 `cases/okr/kr`、`cases/okr/task`、`cases/okr/checkin` 等目录。
   - 每个流程拆分节点：`LoadContext` → `Validate` → `Persist` → `PostHook (progress/log)`。
   - Controller 改为调用对应 Case Service，而非直接调用 domain Service。
3. **测试**：Case 层组合测试验证节点顺序与异常回滚。

## 8. 推荐迭代顺序
| 阶段 | 内容 | 验收要点 |
| ---- | ---- | -------- |
| P0 | KR CRUD + progress 汇总 | Objective progress 自动重算；接口 swagger 可用 |
| P1 | Task CRUD + 用户关联 | “我的任务”列表返回正确；任务状态更新同步写 progress record |
| P2 | Alignment | 对齐树 API 正确；防环校验 |
| P3 | Progress Record | 任意 KR/任务更新后生成历史记录，可分页查询 |
| P4 | Check-in | 复盘创建成功，KR item 记录差值，写入 progress record |
| P5 | Operation Log | 审计接口可查所有关键操作 |
| P6 | Case 编排 | Controller 全部迁移到 case 服务；回归测试通过 |

## 9. 回归建议
- 每阶段完成后运行现有测试 + 新增测试。
- 校验新增 SQL 是否有索引支撑；必要时补充。
- 更新 README/接口文档，保持 swagger 注释与实现同步。
- 与前端同步接口变更，检查 RBAC + 数据权限互锁是否生效。

---
如需进一步细化某一阶段（例如进度记录字段含义、前端交互流程），可以在对应阶段下继续拆分子任务。
